### Experiment conditions
All experiments ran on a single local machine (localhost) with 1000 requests across 50 concurrent workers using Python threads and the `requests` library.

### Local DB
Pros: Quick access, concurrency-safe access using a write lock, very low latency and high throughput with a small number of users. Read operations are not blocked by the write lock — only writers contend — so reads scale better than writes.
Cons: When the number of concurrent users approaches 50 or more, throughput drops and latency increases. This is because each write to the DB acquires a lock, and all other writers must wait for it to release — write p95 jumped from 0.4ms to 354ms at 50 concurrent users in our experiment. GET on `SELECT *` degrades with table size (O(n) full table scan) while indexed lookups like GET by ID stay O(log n) via B-Tree — we observed GET latency climbing as the reservations table grew.
Traded: We traded write concurrency under heavy load for low latency and high throughput at lower user counts. Note: the lock is a concurrency mechanism, not a security feature — it prevents data corruption under simultaneous writes, not unauthorized access.
Reverse: If concurrent writers consistently exceed ~20-30 users, or if the data does not fit naturally into table rows (e.g. images, large files), a different system should be considered.

### S3 Storage
Pros: Highly available, scales to unlimited storage, and handles many concurrent users well. The reason it tolerates more concurrency is that each request is I/O-bound — workers spend most of their time waiting for HTTP responses from S3, so adding more parallel workers keeps more requests in-flight simultaneously. It also stores any file type: images, videos, backups, ML datasets, etc. Best suited for large file storage (ML datasets, images, backups) and batch operations rather than frequent small CRUD.
Cons: 15x slower than SQLite for small structured data (p95 of 140ms vs 0.4ms in our experiment). Has no query capability — no WHERE clause, no indexes, no joins. No concurrency control — race conditions must be handled by the application. No auto-increment — our experiment used a `_counter` object for IDs, which creates a race condition under concurrency where two writers can read the same counter value and overwrite each other.
Traded: Traded low latency and high throughput for small frequent operations, but gained scalable storage and flexibility for large or non-tabular data.
Reverse: If the majority of operations are frequent small reads/writes on structured data under ~1KB, S3 is the wrong tool. Switch to a database when queries, indexes, or transactions are needed.

### Microservice
Pros: Very available, seperation of concerns if two applications server different purposes.
Cons: Comparatively to all other systems, low throughput and high latency due to the need to make more http requests from one service to another before returning a response to the client/user. Latency compounds with each network hop — PUT was the slowest operation (132ms avg) because it makes 2 inter-service HTTP calls (release old slot + book new slot) plus its own DB write, while POST and DELETE each make 1 inter-service call. In our experiment the request mix was: POST 33%, GET 29%, DELETE 16%, PUT 13%, GET availabilities 9% — the throughput number depends heavily on this mix. Note: all services ran on localhost — real network latency would make these numbers significantly worse.
Traded: Traded low latency and high throughput (885 → 559 req/s in our experiment), and introduced distributed consistency risk. Since reservations and availabilities live in two separate databases with no distributed transaction, a crash between the two writes leaves the system in an inconsistent state — for example, if the reservations service crashes after booking an availability slot but before saving the reservation, that slot stays permanently booked. Availability only improves if multiple instances are actually running — a single instance microservice is no more available than a monolith.
Reverse: If the system I implementing is simple enough or server a lower number of clients, then using a microservice architecture is not quite right. For example, if the reservation app is for a very high end restaurant, then a monolith architecture makes more sense - we can keep it simple, and only a small number of clients will ever use the booking system we provide.
